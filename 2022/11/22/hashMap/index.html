java的jdk8的HashMap的put源码解读

1.计算根据key来计算hash值

hash的计算公式为(h = key.hashCode()) ^ (h >>> 16)的原因：

  1. 右移16位是为了让高16位也参与运算，可以更好的均匀散列，减少碰撞，进一步降低hash冲突的几率
  2. 异或运算是为了更好保留两组32位二进制数中各自的特征

2.如果table未创建或长度为0，扩充table

未作设置第一次添加，table长度为16。

3.根据hash计算索引

与运算保证了结果一定在[0,n - 1]范围内，即有效的table下标

4.如果table[i]为空，则直接创建节点赋值

5.如果table[i]不为空，说明需要添加在table[i]的链表或红黑树中，分三种情况查找

5.1 table[i]的的一个节点的Key与要添加的Key相同

首先需要二者hash值相等，由于可能存在哈希冲突的情况，所以相等之后需要去检查是否真的相等：

先检查引用是否相等(对于基本数据类型则是值是否相等)，如果引用不相等，如果equals方法得到true，说明key是重复的。

5.2 table[i]是一颗红黑树

交给红黑树处理

5.3table[i]是遍历

遍历链表，如果找到与key相同的节点，如果找不到，则在最后新建一个节点。如果新建一个节点，还要判新建后断链表长度是否超过TREEIFY_THRESHOLD（jdk8默认值为8），超过后则进入treeifyBin方法。

（添加新节点后的链表长度为binCount + 2，所以添加节点后的长度超过TREEIFY_THRESHOLD，才会进入treeifyBin方法。又意味着如果添加新节点后的链表的长度为TREEIFY_THRESHOLD，那么下一次在该链表添加新节点，就会进入treeifyBin方法）。

注意到代码的if判断，如果tab == null || (n = tab.length) < MIN_TREEIFY_CAPACITY条件成立，则不会进行树化，而是resize()，也就是说当table的长度小于MIN_TREEIFY_CAPACITY则会对table进行扩充，只有当table长度大于等于MIN_TREEIFY_CAPACITY，才会树化。

所以hashMap其树化的条件是一条链表的长度大于TREEIFY_THRESHOLD且talbe的长度大于等于MIN_TREEIFY_CAPACITY。

5.4 如果在map中找到key

完成value的赋值操作。

6. 如果map中的节点数达到阈值，则提前扩充

注意是size达到阈值，size指的是整个table表中的所有的节点的个数。

2到6步操作定义在putVal中，完整的代码如下。

  1. 第一次添加时，table数组扩容到DEFAULT_LOAD_FACTOR(16)，临界值是(int)(DEFAULT_LOAD_FACTOR * DEFAULT_INITIAL_CAPACITY(0.75)) 。
  2. 如果map节点数超过临界值，则扩容为原长度×2，临界值变为扩容后的长度×0.75。
  3. 在 Java8 中，如果一条链表的元素个数超过 TREEIFY_THRESHOLD(默认8)， 并且 table 的大小 >= MIN_TREEIFY_CAPACITY(默认64)，就会进行树化(红黑树)

所以通过源码知道，如果我们需要使用自定义的类，我们希望两个不同的对象在一些属性相同时判断为相同的key，那么需要重写equals和hashCode方法，如果不重写hashCode方法，不同的对象在添加时都会添加进去，只有相同的对象不会添加进去。
